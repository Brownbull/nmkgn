# Pitfalls Research

Generated: 2026-05-12

## Scope

Standard `/gabe-scope` research lane: pitfalls. Focused on what can make this
product unsafe, misleading, or hard to trust.

## Pitfalls

### 1. Becoming an advisor by accident

The product must not tell the user what to do as legal, financial, or accounting
advice. It should produce evidence-backed analysis, questions, options, and
escalation paths. Action language must be careful: "ask", "review", "confirm",
"possible inconsistency", "potential risk".

### 2. Free-form document analysis

The user explicitly rejected a free-form analyzer. If the product accepts any PDF
and asks an LLM to improvise a structure, it violates both scope and the stable
schema rule. Each document type needs its own Pydantic input/output model.

### 3. Before/after signing ambiguity

Before signing and after signing are different products inside the same case
model. Before signing makes market alternatives and negotiation questions more
valuable. After signing makes discrepancy evidence, information requests, record
building, and escalation entities more valuable.

### 4. Market comparison overclaim

CMF notes that simple simulators often estimate operations using core elements,
while real values require additional user-specific information. The app must not
present a benchmark or alternative as the user's guaranteed available offer.

### 5. OCR and extraction errors

Payment count, dates, rates, CAE, and total cost are high-impact fields. They
must be user-confirmable before they drive conclusions. Every extracted
document fact must carry source coordinates and extraction metadata. Extraction
warnings should be part of the schema.

### 6. Legal source drift

Consumer-finance rules, SERNAC materials, CMF pages, and official law text can
change. The app needs source versions or retrieval timestamps for rule catalogs.

### 7. Evidence-less or provenance-less findings

A finding without a clause, calculation, benchmark, or user-provided comparison
input should not be promoted. It should become an uncertainty or a request for
missing context.

The product should also separate claim types:

- `fact`: a value taken from a document or external source, with source
  coordinates or a trackable URL, plus extracted/retrieved dates and verification
  status when verified.
- `inference`: a conclusion generated by AI, marked as inference, with model,
  schema version, run id, generated date, and the facts/sources it used.
- `unsupported_output`: anything that cannot be tied to a source or inference
  trail. This can be stored for audit/debugging as a hallucination candidate, but
  it must not be presented as a trusted finding.

### 8. Privacy and document retention

Users will upload sensitive financial and legal documents. Retention, deletion,
access control, and audit logging are product requirements, not launch polish.

## Product Implications

- The core schema should include `case_stage: before_signing | after_signing`.
- Every extracted fact must have source coordinates, extraction date, and source
  document metadata. External reference facts must include a trackable URL,
  retrieval date, and verification date when verified.
- Every AI inference must be marked as inference and include model, schema
  version, run id, generated date, and evidence inputs.
- Unsupported outputs should be stored as audit/debug records, not shown as
  evidence-backed findings.
- User confirmation should be explicit for high-impact fields.
- Market options should be labeled as benchmarks or comparable references, not
  personalized offers.
- Generated emails should be editable drafts, backed by selected findings.
