# Stack Research

Generated: 2026-05-12

## Scope

Standard `/gabe-scope` research lane: stack. Focused on the technical shape that
supports stable document-specific agents and evidence-backed analysis.

## Recommended Shape

Use a staged pipeline rather than one large agent:

1. Case intake with fixed structured fields.
2. Document upload and OCR/text extraction.
3. Document type detection.
4. Document-specific fact extraction into a stable model.
5. Provenance normalization into fact, inference, and unsupported-output records.
6. User confirmation for high-impact fields.
7. Deterministic calculations and rule-source lookup.
8. Document-specific analysis agent returns a stable Pydantic model.
9. UI displays fields based on analysis plan and case stage.
10. Report/export/email draft generation from selected findings.

## PydanticAI Fit

PydanticAI supports agents with structured output types. The official docs
describe the `Agent` as containing instructions, tools, dependencies, and a
structured output type. They also describe `output_type` as the constructor
argument that makes an agent return a typed structured value.

This fits the product rule:

- `ConsumerCreditAgent` should return `ConsumerCreditAnalysis`.
- Future document agents return their own models.
- The display plan should filter/show fields after generation, not change the
  schema.

Avoid using dynamic JSON schemas as the main contract. PydanticAI's docs note
that custom `StructuredDict` schemas do not get Pydantic validation of received
objects, so BaseModel-style output is safer for v0.

## OCR / Document Processing

Google Document AI's Enterprise OCR documentation supports text and layout
extraction from PDFs and images, with page-level and layout structures. It also
offers features such as native PDF text extraction, rotation correction, image
quality scoring, and checkbox/radio mark extraction. These are relevant because
contract scans can be rotated, low quality, or image-only.

The implementation should not assume one OCR provider forever. Store normalized
document text and source spans behind an internal extraction interface.

## Provenance Contract

Every claim shown to the user should be one of three types:

- `fact`: sourced from an uploaded document or external reference, with source
  coordinates or a trackable URL, extraction/retrieval date, and verification
  date when verified.
- `inference`: generated by AI, explicitly labeled as inference, with model,
  schema version, run id, generated date, and evidence inputs.
- `unsupported_output`: not sufficiently tied to a source or inference trail.
  Store it for audit/debugging as a hallucination candidate, but do not display
  it as a trusted finding.

For uploaded documents, source coordinates should include document id, page,
bounding boxes or text spans, OCR provider, extracted date, and confidence. For
external sources, store the URL that resolves to the referenced content, the
retrieved date, and the verified date when a human or trusted process confirms
it.

## Observability

Track per run:

- upload size and page count
- OCR provider and duration
- extraction confidence and warnings
- agent model and schema version
- run id and generated date for AI inferences
- source URL, retrieval date, and verification date for external references
- token input/output
- latency and cost
- number of findings by severity/confidence
- number of fields requiring user confirmation
